If your Tomcat Manager login throws a 401 error or a 403 error you are not alone and this guide will get you back to deploying like a semi functional sysadmin. We will cover the usual suspects that cause authentication and access failures and show the exact places to look in conf and webapp files so you stop guessing and start fixing.

Quick checklist to stop 401 and 403 responses

  • Add a tomcat user with the manager gui role in conf/tomcat-users.xml
  • Make sure the manager webapp is present and enabled in webapps
  • Check for host restrictions that cause 403 forbidden responses
  • Restart Tomcat so configuration changes take effect
  • Test the manager URL and verify credentials and role membership

Step by step with a hint of sarcasm

Step one is editing the file that actually matters conf/tomcat-users.xml. If you add a user but forget the right role Tomcat will politely refuse login with a 401 error. Add a line like this inside the <tomcat-users> block and yes this file is XML so be careful with syntax.

<user username="deploy" password="s3cr3t" roles="manager-gui" />

Make sure the role name is manager-gui exactly. Tomcat is not going to be forgiving about spaces or synonyms. If you need deployment automation consider creating a dedicated user with only the manager roles you need so you do not hand out root level keys to strangers.

Why a 401 happens

A 401 error means authentication failed. Either the username or password is wrong or the user does not have the manager-gui role. Double check you edited the active conf file. On systems with multiple Tomcat instances or package manager installs the configuration path you edited may not be the one Tomcat actually reads.

Why a 403 happens

A 403 forbidden means authentication passed or was not required but access from your client IP is blocked by a valve or context restriction in the manager webapp. The manager application ships with an access restriction by default so remote hosts are often blocked.

<Context path="/manager" docBase="manager" reloadable="false">
  <Valve className="org.apache.catalina.valves.RemoteAddrValve" allow="127\.0\.0\.1" />
</Context>

If you need remote access add your client address to the allow attribute or remove the valve during controlled testing. Keep access restricted in production and do not open manager GUI to the entire internet. You have been warned.

Restart Tomcat and test login

Tomcat reads conf files at startup. After making changes restart the service so new users and role mappings take effect. On systemd hosts run the usual systemctl command or use bin/catalina.sh stop and start if you prefer drama. Then open your browser and go to localhost slash 8080 slash manager slash html and enter the credentials you added.

Troubleshooting tips that actually help

  • If login still fails verify the user entry and that manager-gui is spelled correctly
  • Check for multiple Tomcat installations and confirm you edited the active conf path
  • Look in catalina logs for authentication errors and context problem messages
  • If the manager webapp is missing check the webapps folder or your package maintainer notes

Recap without the drama. Add a user with manager-gui in conf/tomcat-users.xml. Make sure the manager webapp allows your client address or is enabled locally. Restart Tomcat and test the manager URL. Do that and your 401 and 403 errors should vanish like bad coffee from the break room.
