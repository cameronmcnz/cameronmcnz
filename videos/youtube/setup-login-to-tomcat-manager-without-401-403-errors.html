Quick summary for the impatient server admin

401 means Tomcat does not like your credentials. 403 means Tomcat does not like where you are coming from. If you want the manager GUI to stop yelling at you follow these steps and do not use passwords like password123 unless you enjoy chaos.

Add a manager user the right way

Edit conf/tomcat-users.xml and add a role and a user entry for the manager GUI. A minimal example looks like this

<role rolename="manager-gui" />
<user username="admin" password="YourStrongPasswordHere" roles="manager-gui" />

Make the password strong and never commit that file to a public repo. If your Tomcat is managed by a package or automation tool check that tool first so your changes do not get overwritten.

Restart Tomcat and reload the realm

You must restart the server so the realm picks up tomcat-users.xml. Use systemctl restart tomcat if you installed Tomcat as a service or use bin/shutdown.sh and bin/startup.sh for manual installs. A restart is the usual reason people still see 401 errors after they added a user.

Why 401 keeps happening

  • No matching user or wrong password. Double check the username and role names.
  • The user lacks the manager-gui role. Manager needs that role to grant access to the GUI.

Check manager webapp restrictions and 403 causes

Some Tomcat distributions block the manager GUI to remote hosts by default. Look for a RemoteAddrValve in webapps/manager/META-INF/context.xml or conf/Catalina/localhost/manager.xml. If there is a rule only allowing 127.0.0.1 that will give you a 403 when you try to connect from elsewhere.

Network gotchas

  • If Tomcat is bound to a specific address or running behind a reverse proxy the request remote address may not be what Tomcat expects. That can trigger a 403.
  • When using a proxy ensure RemoteIpValve is configured so Tomcat can read the real client IP from standard headers.

Accessing the Manager GUI

Point your browser to http://localhost:8080/manager/html on the server host or use your server hostname and port. Enter the username and password you added. If you still get a 401 check catalina.out and the manager logs for realm errors.

Final hardening tips

  • Restrict manager access to trusted hosts with a RemoteAddrValve rather than leaving it open to the internet.
  • Do not reuse your manager credentials in applications or source control.
  • If using a reverse proxy require client auth or IP restrictions there as well.

Follow these steps and you will resolve most Tomcat Manager 401 and 403 problems. If trouble persists read the logs and recheck any proxy or firewall rules that may rewrite headers or mask client addresses.
