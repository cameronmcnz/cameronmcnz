Welcome to the magic closet where plain markdown becomes a full blown static site and where front matter holds the keys. If you like predictable templates and slightly smug control over your content then this guide will show you how to use Jekyll front matter with Liquid objects tags and filters without summoning dark template voodoo.

Front matter basics

Front matter is the YAML block at the top of a page or post that tells Jekyll who you are and what to do with the file. It sits between three dashes and exposes variables to your layouts and includes. Be tidy with names and you will avoid debugging that feels like arguing with a toaster.

---
title: My Post
date: 2025-01-01
tags:
  - jekyll
  - tutorial
layout: post
---

Common variables are title date tags and layout. Use consistent keys so your templates do not silently fail when you typo a variable and then cry into the build log.

Liquid objects you will actually use

Liquid gives you a few objects that are worth memorizing. The page object contains front matter for the current file. The site object contains site wide collections pages and posts. Jekyll also builds helpers like site.posts and site.tags which can save you a lot of looping.

Looping and grouping posts by tag

Jekyll creates site.tags as a map where the key is the tag name and the value is the array of posts with that tag. This is the easiest and fastest way to make a tag index. No need to reinvent the wheel.

{% raw %}{% for tag in site.tags %}
  {% assign tag_name = tag[0] %}
  {% assign posts = tag[1] %}
  
{{ tag_name }} ({{ posts | size }})

{% endfor %}{% endraw %}

If you prefer to iterate over all posts and build a custom structure you can do that too. But site.tags is the built in shortcut and it behaves well.

Filters to shape output

Filters change presentation without changing your data. Chain them to tidy things up. Some useful ones are date where sort join uniq map and size. Order matters so read left to right and swap them if the output looks suspicious.

Build locally and test often

Run a local server to preview changes. Iteration is faster than guessing. If something breaks check for typos in variable names or inconsistent tag casing. GitHub Pages has a limited plugin set so test builds locally with the same configuration you deploy with.

  • Use consistent tag names and casing
  • Prefer site.tags for tag indexes
  • Chain filters to refine results
  • Run the local server and check templates after each change

Final tip that sounds obvious but is not

Templates are powerful because they keep content and presentation separate. Front matter turns a markdown file into structured data that Liquid can read and act on. Learn a few Liquid filters and objects and your static site will do a lot more heavy lifting while you sip coffee and pretend you meant to write that CSS.
